# RabbitMQ Operational Parity Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete the operational RabbitMQ Management workflows defined in the approved parity design without adding a RabbitLens backend.

**Architecture:** Extend the existing typed domain/query/feature boundaries. A session-scoped capability model controls optional navigation and direct-route behavior; protocol-, queue-, and plugin-specific UI consumes typed Management API clients and reusable operational components. Each milestone follows red-green-refactor and leaves the application independently buildable.

**Tech Stack:** React 19, TypeScript 6, TanStack Router/Query/Table, Zod 4, React Hook Form, Radix/shadcn, Tailwind CSS 4, Vitest, Testing Library, MSW, Playwright, Docker Compose RabbitMQ.

## Completion gate

Do not call the project operationally complete unless all of the following are freshly true:

- Capability discovery hides unavailable plugins and distinguishes absence from forbidden access.
- AMQP 0-9-1 and AMQP 1.0 connection details choose the correct child resources.
- Queue/channel consumers and quorum/stream replication state are visible.
- Queue publish, supported actions, and capability-gated Move Messages work.
- Operator-focused node persistence, I/O, and churn diagnostics render only available data.
- Super-stream behavior is truthful to the installed plugin: RabbitMQ exposes create via Management UI assets, so RabbitLens implements create-only and does not invent list/detail/delete controls.
- The parity verifier covers actions and data sections rather than only pages.
- Full frontend, browser, bundle, secret, and integrated RabbitMQ checks pass with fresh output.
